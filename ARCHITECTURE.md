## 1. Overview

The system is an **SRE Incident Triage Platform** built on **Hexagonal Architecture (Ports & Adapters)**, orchestrated by a **LangGraph state machine**, and packaged as a Docker Compose stack.

### 1.1 Platform structure (v4 — two-tier)

The platform is composed of two logical tiers:

**Platform layer (new in v4):**
- `sre-web` — React 18 SPA (TypeScript + Vite + Tailwind + SoftServe Design System) served via Nginx. Provides the full platform UI: authentication, incident management, configuration, dashboard.

**Agent layer (original):**
- `sre-agent` — FastAPI backend. Hosts the LangGraph agent pipeline, all domain ports, adapters, the REST API (consumed by `sre-web`), and the Prometheus `/metrics` endpoint.
- `mock-services` — GitLab-Issues-compatible ticket and notification mock (default adapter, swappable via DB config).
- `app-db` — PostgreSQL for incident persistence, RBAC users, LLM config, platform config, governance thresholds (all in separate tables, same DB).
- `langfuse-*` — self-hosted Langfuse observability stack.

### 1.2 Platform overview

**Five containers in total (v4):**

1. **`sre-web`** — React SPA (Nginx). Exposed on host port `5173`. No Python, no business logic. Calls `sre-agent` API at `http://localhost:8000`. *(NEW — v4)*
2. **`sre-agent`** — FastAPI + LangGraph + all adapters. Exposed on host port `8000`. Now also serves: auth endpoints, RBAC middleware, platform config API, LLM hot reload, eShop indexing.
3. **`mock-services`** — GitLab-Issues-compatible mock. Exposed on host port `9000`.
4. **`app-db`** — PostgreSQL (internal only). Stores: incidents, tickets, files, governance thresholds, users, llm_config, platform_config.
5. **`langfuse-*`** — Self-hosted Langfuse. `langfuse-web` exposed on host port `3000`.

**Host-exposed ports (v4):** `5173` (sre-web), `8000` (sre-agent), `9000` (mock-services), `3000` (Langfuse). See ARC-020 and DEC-A10 for port governance.

### 1.3 Architectural pattern: **Hexagonal Architecture + LangGraph orchestration**

The **domain** (business logic of incident triage) does not know that LLMs, HTTP, databases, FAISS, or LangGraph exist. It depends only on **ports** (Python ABCs). Concrete **adapters** implement those ports and are wired at startup by `infrastructure/container.py` based on environment variables and — from v4 — DB-stored configuration.

**LangGraph remains the orchestrator** of the 6-stage pipeline. The React SPA is a pure consumer of the FastAPI REST API — it never calls adapters directly.

**Why this pattern:**
- Same Docker Compose runtime as v1 (minimal host ports, single deploy unit) — **no operational overhead added**
- Swap mock → production = DB config change, **zero code change**
- Domain is unit-testable without booting FastAPI, LangGraph, or any adapter
- Resilience patterns (circuit breaker, fallback, retry) live in *one* place per port — not scattered
- React SPA is decoupled from the Python backend — can be developed and built independently

**Rejected alternatives:**

| Pattern | Why rejected |
|---|---|
| Pure modular monolith (v1) | Business logic tangled with framework calls. Cannot swap LLM/ticketing without editing nodes. |
| Microservices per stage | 48h is not enough wiring time. No scaling need at hackathon scale. |
| FastAPI serving React build (DEC-A01 option b) | Couples Node build tooling into Python container. Complicates Dockerfile and CI. Nginx is the right tool for serving SPAs. |
| Workflow engine (Temporal/Airflow) | Operational overhead. LangGraph already gives a typed DAG with native tracing. |

## 4. Observability Contract v2

> Source of truth for `@developer`. v2 (2026-04-07) restructures observability into **3 layers**: infra spans (per stage), LLM attributes (per call), and Agent behavior attributes (per agent invocation). All three feed Langfuse for traces and Prometheus for metrics.
>
> See [`docs/diagrams/observability-layers.md`](docs/diagrams/observability-layers.md).

### 4.1 Layer 1 — Infra spans (per stage)

Span naming convention: `sre.agent.<name>.<phase>` (e.g. `sre.agent.triage.analysis`).

| Stage | Span name | Required attributes |
|---|---|---|
| 1. Ingest | `sre.agent.intake_guard.ingest` | `incident_id`, `has_image: bool`, `has_log: bool`, `text_length: int` |
| 2. Guardrails | `sre.agent.intake_guard.guardrails` | `incident_id`, `injection_detected: bool`, `score: float`, `blocked_reason?: str` |
| 3. Triage | `sre.agent.triage.analysis` | `incident_id`, `context_docs: list[str]`, `severity: str` (+ all Layer 2 attrs + all Layer 3 attrs) |
| 4. Ticket | `sre.agent.integration.ticket_create` | `incident_id`, `ticket_id: str`, `ticket_provider: str`, `severity: str` |
| 5. Notify Team | `sre.agent.integration.notify_team` | `incident_id`, `ticket_id`, `notify_provider: str`, `recipients_count: int` |
| 6. Resolve Notify | `sre.agent.resolution.notify_reporter` | `incident_id`, `ticket_id`, `reporter_email: str` |

All spans share the same root trace identified by `incident_id`. Langfuse `session_id = incident_id`.

### 4.2 Layer 2 — LLM attributes (per LLM call)

Every span that wraps a call to `ILLMProvider` MUST emit:

| Attribute | Type | Description |
|---|---|---|
| `llm.cost_usd` | float | Estimated cost = tokens × provider price |
| `llm.prompt_name` | str | Logical prompt id from `PROMPT_REGISTRY` |
| `llm.prompt_version` | str | Semver of the prompt template (e.g. `1.0.0`) |
| `llm.tokens_in` | int | Input token count |
| `llm.tokens_out` | int | Output token count |
| `llm.provider_used` | str | Which provider actually responded (`gemini`, `openrouter`, …) |
| `llm.fallback_used` | bool | True if circuit breaker fell over to fallback |

### 4.3 Layer 3 — Agent behavior attributes (per agent invocation)

Every agent subgraph MUST emit a root span with:

| Attribute | Type | Description |
|---|---|---|
| `agent.name` | str | One of `intake_guard`, `triage`, `integration`, `resolution` |
| `agent.iterations` | int | Loop iterations consumed (ReAct steps) |
| `agent.max_iterations` | int | Configured upper bound |
| `agent.rag_queries` | int | Number of `IContextProvider.search_context` calls |
| `agent.rag_hits` | int | Number of context docs returned (sum across queries) |
| `agent.escalated` | bool | True if `should_escalate` returned True |
| `agent.tool_calls` | list[str] | Names of tools invoked (`search_context`, `create_ticket`, …) |

### 4.4 Quality signals (orchestrator root span)

The orchestrator's root span (one per incident) MUST emit, in addition to all child attributes:

| Attribute | Type | Description |
|---|---|---|
| `triage.severity` | str | `CRITICAL` \| `HIGH` \| `MEDIUM` \| `LOW` |
| `triage.confidence` | str | `high` \| `medium` \| `low` |
| `triage.quality_score` | float | 0.0–1.0, populated by LLM-judge in eval mode |
| `triage.affected_components` | list[str] | Components identified by triage |
| `case.status_final` | str | Terminal `CaseState.status` |
| `case.total_duration_ms` | int | Wall clock from ingest to terminal status |

### 4.5 Evaluation Layer

> See [`docs/diagrams/eval-pipeline.md`](docs/diagrams/eval-pipeline.md).

The system ships with an offline **LLM-as-judge evaluation pipeline** that runs against a curated golden dataset on every PR (CI hard gate, see ARC-016).

**Components:**
- **Golden dataset** — `services/sre-agent/evals/golden_dataset.jsonl` — curated cases with expected outcomes
- **Eval runner** — invokes the Triage agent through `ILLMProvider` (no shortcuts) for each case
- **TriageJudge** — LLM-as-judge that scores each output against the expected
- **Langfuse Dataset Experiment** — every run is logged as a Dataset Run in Langfuse for trend analysis
- **CI gate** — exit code non-zero if thresholds are breached → PR blocked

**Golden case schema:**
```json
{
  "id": "golden-NNN",
  "description": "...",
  "input": {
    "text": "...",
    "modality": "text|log|image",
    "reporter_email": "..."
  },
  "expected": {
    "severity": "CRITICAL|HIGH|MEDIUM|LOW",
    "affected_components": ["..."],
    "root_cause_keywords": ["..."],
    "should_block": false
  }
}
```

**EvalScore schema (returned per case):**
```python
@dataclass(frozen=True)
class EvalScore:
    case_id: str
    severity_correct: bool
    severity_score: float       # 1.0 if correct else 0.0
    root_cause_score: float     # keywords_found / keywords_expected
    components_score: float     # jaccard(expected, predicted)
    overall_score: float        # 0.4*severity + 0.4*root_cause + 0.2*components
    judge_reasoning: str
    blocked_correctly: bool | None   # only for adversarial cases
```

**CI thresholds (hard gate, ARC-016):**

| Metric | Threshold | Failure mode |
|---|---|---|
| `avg_overall_score` | `>= 0.70` | CI fails → PR blocked |
| `adversarial_recall` | `== 1.0` (100%) | CI fails → PR blocked |

The eval runner is owned by `@developer`. The thresholds and golden dataset shape are owned by `@architect` — modifying either requires architect approval.

> **NOT IMPLEMENTED IN v4 — deferred to post-hackathon v5.**

**New entities (owned by `app/domain/entities/explainability.py`):**
```python
class RagAttribution(BaseModel):
    doc_id: str
    chunk_preview: str        # first 200 chars of the chunk content
    relevance_score: float    # 0.0–1.0; higher = more relevant to the triage output
    contributed_to: list[str] # claim IDs this doc supported (from OutputGrounding)

class OutputGrounding(BaseModel):
    claim: str                 # one factual claim extracted from triage summary
    grounded: bool             # True if at least one RAG doc supports this claim
    source_doc_id: Optional[str]   # doc_id of the primary supporting doc (if grounded)
    grounding_score: float     # 0.0–1.0; confidence that the claim is grounded

class ExplainabilityReport(BaseModel):
    incident_id: str
    rag_attributions: list[RagAttribution]
    output_groundings: list[OutputGrounding]
    overall_grounding_score: float   # mean(grounding_score) across all claims
    generated_at: datetime

class FeedbackRecord(BaseModel):
    incident_id: str
    rating: Literal["positive", "negative", "neutral"]
    comment: Optional[str] = None
    submitted_by: Optional[str] = None     # operator email or None for anonymous
    submitted_at: datetime
```

**`TriageResult` extension (edit `app/domain/entities/triage.py`):**
```python
# Add to TriageResult:

```

> An empty `rag_attribution` list is ONLY acceptable when `TriageResult.degraded == True` (ARC-018).

**`IExplainabilityProvider` port contract (owned by `app/domain/ports/explainability_provider.py`):**
```python
class IExplainabilityProvider(ABC):
    name: str  # "langfuse" | "memory"

    async def compute_report(

        Called by the orchestrator after the Triage agent returns its TriageResult.
        Must NOT raise — on LLM failure, returns a degraded report with
        overall_grounding_score=0.0 and empty output_groundings.
        """
        ...

    async def post_feedback(self, feedback: FeedbackRecord) -> None:
        """Posts a human SRE feedback record to Langfuse as an online score.
        Score name: 'human_feedback'. Value: 1.0 (positive) | 0.5 (neutral) | 0.0 (negative).
        Comment stored as score comment. Fire-and-forget in production — log on failure.

    async def post_online_eval(

        """Posts the incident as a Langfuse Dataset Run for online eval tracking.
        Scores emitted: grounding_score, confidence_score.
        Called as a background task after the orchestrator root span closes (ARC-019).
        In memory_adapter: always silently drops (test isolation).
        """
        ...
```

### 4.14 Config-from-DB Contract (v4)

> **Principle (established v4 — 2026-04-08):** After initial bootstrap (DB connection, app environment, Langfuse host), all system behavior is governed by database-stored configuration managed through the platform UI (`/config`). `.env` contains only infrastructure bootstrap variables. No business logic value may be hardcoded in Python files or `.env` after HU-P032 is implemented.
>
> This contract is the source of truth for `@developer` implementing the config layer and `@architect` reviewing compliance.

#### 4.14.1 Bootstrap variables (.env ONLY — never in DB, never in UI)

| Variable | Reason — why it cannot live in the DB |
|---|---|
| `APP_DATABASE_URL` | Required before the DB is reachable — bootstraps the DB connection itself |
| `APP_ENV` | Determines startup validation behavior (`_reject_stub_in_production`) — must be available before DB read |
| `LANGFUSE_HOST` | Required to connect to Langfuse before DB config loads |
| `SRE_API_KEY` | Legacy backward-compatibility key during JWT migration |
| `MOCK_SERVICES_URL` | Infrastructure container address — not business config |
| `JWT_SECRET` | Cryptographic secret — must never be stored in the DB it protects |
| `CONFIG_ENCRYPTION_KEY` | AES-256 key used to encrypt credentials stored in the DB — cannot be in the DB it encrypts |

#### 4.14.2 DB-managed configuration — table structure (DEC-A04)

**Decision DEC-A04:** Separate port structure — `IGovernanceProvider` keeps exclusive ownership of governance thresholds (`governance_thresholds` table). Two new ports are created:
- **`ILLMConfigProvider`** — owns the `llm_config` table (LLM provider, model, API keys, circuit breaker params)
- **`IPlatformConfigProvider`** — owns the `platform_config` table (all other business variables: ticket, notify, context, security, observability, storage)

Rationale: single-responsibility per port, no breaking change to existing `IGovernanceProvider` (HU-012 implementation plan unchanged), clean separation between governance business logic and infrastructure adapter configuration.

**Three DB tables (see §11 for full SQL schemas):**

| Table | Port | Variables managed | Hot reload? |
|---|---|---|---|
| `governance_thresholds` | `IGovernanceProvider` | 5 governance thresholds (confidence, quality score, severity threshold, kill switch, max RAG docs) | Yes — TTL cache invalidated on PUT |
| `llm_config` | `ILLMConfigProvider` | Provider, model, API keys (encrypted), fallback provider/model, circuit breaker params | Yes — < 5 seconds (DEC-A06) |
| `platform_config` | `IPlatformConfigProvider` | Ticket provider + credentials, notify provider + credentials, context source, eShop config, security params, observability params, storage backend, allowed file types | Partial — most hot-reload; eShop URL requires restart |

#### 4.14.3 Cold-start seed strategy (DEC-A05)

**Strategy:** Alembic migration seed — idempotent `INSERT ... ON CONFLICT DO NOTHING` for all rows in all three config tables. The seed runs as part of `alembic upgrade head` (already executed in the `sre-agent` container startup command).

**Migration structure:**
- Existing migration: seeds `governance_thresholds` (5 rows) — no change
- New migration A: creates `llm_config` table + seeds default primary/fallback rows
- New migration B: creates `platform_config` table + seeds all keys with documented defaults
- New migration C: creates `users` table + seeds 5 demo users (one per role) for hackathon

**Behavior when DB is unavailable at startup:**
1. The FastAPI app fails to start — `alembic upgrade head` exits non-zero.
2. Docker Compose `depends_on: app-db: condition: service_healthy` prevents startup before Postgres is ready.
3. If the DB becomes unavailable AFTER startup: `IGovernanceProvider`, `ILLMConfigProvider`, and `IPlatformConfigProvider` each return their in-memory cached values. If no cache exists, they return Pydantic model defaults. They MUST NOT raise — see ARC-020.

**Rule:** The system must never fail to triage an incident because the config DB is temporarily unavailable. All providers must have safe in-memory defaults.

#### 4.14.4 LLM hot reload mechanism (DEC-A06)

**Decision:** Option (c) — **cache with explicit TTL invalidation**. The `container.py` holds a singleton reference to the active `LLMCircuitBreaker`. An `asyncio.Lock`-protected `reload_llm_adapter()` coroutine replaces the adapter atomically.

**Mechanism:**
1. `PUT /config/llm` saves to `llm_config` table.
2. The route handler calls `container.reload_llm_adapter()` after a successful DB write.
3. `reload_llm_adapter()` acquires an `asyncio.Lock`, instantiates the new primary and fallback adapters from the just-saved config, wraps them in a new `LLMCircuitBreaker`, and atomically replaces the singleton reference.
4. Concurrent requests that started before the reload complete with the old adapter (safe — atomically replaced reference). New requests after the lock releases use the new adapter.
5. Total propagation time: < 1s (no polling needed). SLO: < 5s end-to-end (AC-06 / SA-012).

**Circuit breaker reconfiguration (DEC — circuit breaker hot reload):**
The `LLMCircuitBreaker` must expose `reconfigure(threshold: int, cooldown: int)` method that updates the parameters in-place **without losing the current failure count**. If the circuit breaker is currently `OPEN`, it continues open until the previous cooldown expires, then uses the new cooldown for subsequent states.

**Rule (ARC-020-derived):** The `reload_llm_adapter()` call is the ONLY mechanism for changing the active LLM adapter. No other code path may replace the adapter. The route handler is responsible for calling it after a successful DB write.

#### 4.15.3 JWT structure

JWT signed with HS256 using `JWT_SECRET` env var. Token lifetime: 24 hours.

```json
{
  "sub": "user@example.com",
  "role": "operator",
  "name": "Full Name",
  "iat": 1712600000,
  "exp": 1712686400
}
```

**Claims:**
- `sub` — user email (must match a row in the `users` table where `is_active = true`)
- `role` — one of `superadmin`, `admin`, `flow_configurator`, `operator`, `viewer`
- `name` — display name from the `users` table
- `iat` / `exp` — standard JWT timestamps

#### 4.15.4 Mock Google OAuth flow (hackathon)

The system simulates Google OAuth without a real Google Cloud project.

**Endpoint:** `GET /auth/google/callback?email={email}`

**Flow:**
1. React login page shows a dropdown with 5 pre-seeded demo users (one per role).
2. User selects a demo user and clicks "Sign in with Google".
3. React calls `GET /auth/google/callback?email={selected_email}`.
4. FastAPI looks up the email in the `users` table.
5. If found and `is_active=true`: returns `{"access_token": "...", "token_type": "Bearer"}`.
6. If not found or inactive: returns HTTP 401 `{"detail": "user_not_authorized"}`.
7. React stores the token in **`localStorage`** (DEC-A02 decision — see §15).
8. All subsequent API calls include `Authorization: Bearer {token}` header.

**JWT storage — DEC-A02:** `localStorage`. Rationale: for the hackathon mock auth context, the simplicity and CSRF immunity of `localStorage` outweigh the XSS risk. The mock auth does not protect real credentials. A production deployment would migrate to `httpOnly` cookies with `SameSite=Strict`. This is documented as a known security trade-off (SA-003).

#### 4.15.5 RBAC enforcement implementation

```python
# app/api/deps.py — FastAPI dependency

from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
import jwt  # python-jose

security = HTTPBearer()

async def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(security)
) -> dict:
    """Decode and validate JWT. Returns payload dict."""
    ...

def require_role(*roles: str):
    """Returns a FastAPI dependency that enforces role membership."""
    async def _dep(user: dict = Depends(get_current_user)) -> dict:
        if user["role"] not in roles:
            raise HTTPException(status_code=403, detail="insufficient_role")
        return user
    return _dep

# Usage in route:
# @router.put("/config/llm")
# async def update_llm_config(
#     _: dict = Depends(require_role("admin", "superadmin")),
#     ...
# ):
```

**Rule (ARC-022):** RBAC is enforced at the FastAPI route level via `require_role()`. The domain layer has no knowledge of roles. The orchestration layer has no knowledge of roles. Putting role checks in agents, services, or adapters is a violation.

## 5. Architecture Rules — DO NOT BREAK

These are enforced by `@architect` review. Any violation fails the gate.

1. **Domain purity.** Files under `app/domain/` MUST NOT import anything outside the standard library, `pydantic`, or other `app/domain/` files. No FastAPI, no LangGraph, no httpx, no SDKs.
2. **Agents depend on ports, not adapters.** Files under `app/orchestration/agents/` and `app/orchestration/orchestrator/` MUST import from `app/domain/ports/` only — never from `app/adapters/`. The `container` injects concrete adapters.
3. **The orchestrator graph in `app/orchestration/orchestrator/graph.py` is the ONLY top-level coordinator.** API routes invoke `build_orchestrator_graph()` (sync flow) or `build_resolution_graph()` (async webhook flow); they never coordinate agents directly.
4. **Adapters are leaves.** Files under `app/adapters/` MUST NOT import from `app/orchestration/`, `app/api/`, or other adapter packages. They depend on `app/domain/ports/` (to implement) and external libraries.
5. **Every stage MUST emit a Langfuse span and a structured log line** with `incident_id`. (See section 4.)
6. **No LLM call may use raw user input as instructions.** All user input passes through `app/security/` first AND is wrapped in clearly delimited content blocks.
7. **Tool/port calls accept only typed Pydantic args.** No free-form URLs, shell strings, or filenames flow from LLM into adapter execution.
8. **Secrets only via environment variables.** Never commit `.env`. Never hardcode keys.
9. **No new dependencies without `@architect` approval.** All deps live in `services/sre-agent/requirements.txt`.
10. **Only the host ports declared in `docker-compose.yml` may be exposed.** Adding ports requires `@architect` approval. Today: `8000` (sre-agent), `9000` (mock-services), `3000` (Langfuse). `app-db` is internal-only.
11. **Adapter selection is config, never code.** A new provider is added by writing a new adapter class and registering it in `infrastructure/container.py` — never by editing nodes, services, or the graph.
12. **Output schemas are Pydantic models.** LLMs return structured output via the `ILLMProvider.triage()` contract. Never parsed regex.
13. **(ARC-012) Single writer for CaseState.** Only the orchestrator may mutate `CaseState`. Agents receive an immutable Projection (`IntakeProjection`, `TriageProjection`, `IntegrationProjection`, `ResolutionProjection`) and MUST return exactly one `AgentEvent`. The orchestrator folds the event into CaseState. This guarantees a single source of truth and a replayable audit log.
14. **(ARC-013) Agents are subgraphs, not free-form code.** Each agent under `app/orchestration/agents/<name>/` exposes a `build_<name>_agent(container)` factory that returns a compiled LangGraph. The orchestrator composes them via `add_node` — never by importing internal agent functions.
15. **(ARC-014) Resolution is async-only.** The Resolution Agent MUST NOT be wired into the synchronous incident graph. It is exposed via `build_resolution_graph()` and triggered by the resolution webhook handler. Rationale: the sync API response must never block on a human/external resolution event (DEC-006).
16. **(ARC-015) Prompt registry is mandatory.** Every LLM prompt MUST be resolved through `app/llm/prompt_registry.py::PROMPT_REGISTRY`. Inline prompt strings inside `app/orchestration/agents/**` are **forbidden**. Each registry entry carries a versioned id (`<purpose>-v<major>.<minor>.<patch>`) which is emitted as `llm.prompt_name` / `llm.prompt_version` on every LLM span (see §4.2). Reviewer rejects on sight any agent file containing a multiline string passed to `ILLMProvider`.
17. **(ARC-016) Evals are a CI hard gate.** The eval runner (see §4.5) MUST execute on every pull request via GitHub Actions. A PR MUST be blocked from merge if `avg_overall_score < 0.70` OR if any adversarial case is not blocked (`adversarial_recall < 1.0`). The CI job `eval` is non-skippable. Threshold changes require `@architect` approval and a corresponding update to this section.

18. **(ARC-017) Governance thresholds are runtime config, not code.** Business governance thresholds (`confidence_escalation_min`, `quality_score_min_for_autoticket`, `severity_autoticket_threshold`, `kill_switch_enabled`, `max_rag_docs_to_expose`) MUST be stored in the `platform_config` table under `section="governance"` and accessed exclusively via `IPlatformConfigProvider.get_config("governance")`. The API layer pre-loads governance into `CaseState["governance"]` before graph invocation to keep `should_escalate()` a pure function (ARC-014). Hardcoding any threshold as a Python literal in nodes, agents, routers, or services is **forbidden**. The only acceptable hardcoded threshold is the CI eval gate (`avg_overall_score >= 0.70`), documented in §4.5.

19. **(ARC-018) Explainability data — deferred to v5.** `IExplainabilityProvider`, `RagAttribution`, `OutputGrounding`, and `ExplainabilityReport` are not implemented in v4. `TriageResult.rag_attribution` does not exist in the v4 entity. When v5 is implemented, every triage result MUST carry explainability data and an empty `rag_attribution` list is only acceptable when `TriageResult.degraded == True`.

20. **(ARC-019) Online eval — deferred to v5.** `IExplainabilityProvider.post_online_eval()` is not implemented in v4. The `sre_online_eval_post_failures_total` Prometheus counter is defined as a placeholder. When v5 is implemented, `post_online_eval()` MUST fire as a background task in production, with errors logged and counted via that counter.

21. **(ARC-020) No business configuration value may be hardcoded after HU-P032.** After the Config-from-DB platform is deployed, no business configuration value (LLM provider, API keys, ticket system, notify provider, file type allowlists, circuit breaker thresholds, observability settings, security settings) may be hardcoded as a Python literal in source code or as a value in `.env` files. All such values MUST live in one of the two active config tables (`llm_config`, `platform_config`) — governance thresholds are a section of `platform_config` — and be accessed via the corresponding port (`ILLMConfigProvider`, `IPlatformConfigProvider`). The ONLY permitted `.env` values are the seven bootstrap variables listed in §4.14.1. Config providers MUST return safe in-memory defaults when the DB is unreachable — they MUST NOT raise.

22. **(ARC-021) React UI text is English-only. SoftServe Design Tokens are the single styling source of truth.** All user-visible strings in the React SPA (labels, placeholders, validation messages, button text, toast messages, section headers, error messages) MUST be in English. No i18n library (react-i18next, lingui, FormatJS) may be installed or used. No locale files (`.json` translations) may be created. All visual styling MUST use the SoftServe Design System tokens documented in §10 — no hardcoded color hex values, no arbitrary Tailwind classes that override the design token palette. Reviewer rejects on sight any Spanish text in React component output or any `text-[#hex]` Tailwind class that bypasses the token system.

23. **(ARC-022) RBAC is enforced exclusively at the FastAPI route layer.** The `require_role(*roles)` dependency defined in `app/api/deps.py` is the ONLY mechanism for enforcing role-based access control. Domain entities, domain services, domain ports, orchestration agents, and adapters MUST NOT contain role-checking logic. UI-level role hiding (hiding buttons, sections, or routes in React) is PERMITTED as a UX enhancement but is NOT a security control — the backend enforcement is authoritative. Reviewer rejects on sight any `if user.role == "admin"` conditional inside `app/domain/`, `app/orchestration/`, or `app/adapters/`.

> Source of truth for the multi-agent topology. See `docs/diagrams/multi-agent-topology.md`.

### 6.2.1 Topology

The system is composed of **four agents** coordinated by a thin orchestrator. Each agent is a compiled LangGraph subgraph.

```
                ┌──────────────────────── Orchestrator (CaseState owner) ────────────────────────┐
                │                                                                                │
POST /incidents ─►  intake_guard  ──►  triage  ──►  integration  ──►  END                        │
                │       (sub)         (sub)         (sub)                                        │
                │                                                                                │
                │   [resolution graph — separate compiled flow, async]                           │
POST /webhooks/ ─►  resolution                                                                   │
  resolution    │       (sub)                                                                    │
                └────────────────────────────────────────────────────────────────────────────────┘
```

| Agent | Pattern | Tools | Purpose |
|---|---|---|---|
| **IntakeGuard** | Deterministic + LLM judge (5 layers) | none | PII / injection / off-topic gate |
| **Triage** | ReAct loop | `search_context` | Classify + produce `TriageResult` |
| **Integration** | Deterministic 2-step | `create_ticket`, `notify_team` | Persist ticket + page on-call |
| **Resolution** | Deterministic 2-step (async) | `notify_reporter`, `persist_status` | Close the loop with the reporter |

### 6.2.2 State Contracts

Two distinct state types — they MUST NOT be confused:

| Type | Owner | Lifetime | Mutability |
|---|---|---|---|
| **`CaseState`** (`orchestrator/state.py`) | Orchestrator only | One incident, full lifecycle | Mutable, single writer |
| **`AgentLocalState`** (`shared/agent_state.py`) | One agent subgraph | One subgraph invocation | Mutable inside the subgraph, dies on return |
| **`<X>Projection`** (frozen dataclass in `orchestrator/state.py`) | Built by orchestrator, read by agent | One subgraph invocation | **Immutable** |
| **`AgentEvent`** (frozen dataclass) | Returned by agent, applied by orchestrator | Persisted in `CaseState.events` | **Immutable** |

**Flow:**
1. Orchestrator builds `XProjection` from `CaseState`.
2. Orchestrator invokes agent subgraph with `{"projection": proj}`.
3. Agent runs internally on its own `AgentLocalState`.
4. Agent returns `final_event: AgentEvent`.
5. Orchestrator folds `AgentEvent` into `CaseState` (the **only** mutation point).

### 6.2.3 CaseState lifecycle

See `docs/diagrams/state-case-lifecycle.md`. Status enum:

```
NEW → INTAKE_OK → TRIAGED → TICKETED → NOTIFIED → AWAITING_RESOLUTION → RESOLVED
  │
  └─► INTAKE_BLOCKED   (terminal)
  └─► FAILED            (terminal, on agent.error)
```

### 6.2.4 Escalation policy

Lives in `orchestrator/router.py` — pure functions only, no I/O. To change routing logic, edit this file and only this file.

| Routing decision | Function | Current policy |
|---|---|---|
| After intake | `route_after_intake` | `INTAKE_BLOCKED` → END, otherwise → triage |
| After triage | `route_after_triage` | `error` → END, otherwise → integration |
| After integration | `route_after_integration` | always → END (resolution is async) |
| `should_escalate(state, thresholds)` | governance-driven (ARC-017) | `kill_switch_enabled` → `True, "kill_switch"` \| `confidence < confidence_escalation_min` → `True, "low_confidence"` \| otherwise → `False, None` |

> **ARC-017 compliance:** `should_escalate()` signature changed from `(state)` to `(state, thresholds: GovernanceThresholds)`. The orchestrator fetches thresholds via `IGovernanceProvider.get_thresholds()` once at graph entry and passes the result to all routing functions that require it. Hardcoding threshold values in `router.py` is a rule violation.

### 6.2.5 Why two compiled graphs (DEC-006 / ARC-014)

The synchronous incident graph (`build_orchestrator_graph`) returns within bounded time — its SLO is API response latency. Resolution depends on external human/system action of unknown duration, so it lives in a separate graph (`build_resolution_graph`) invoked by the resolution webhook handler. The two graphs share `CaseState` schema but never share a single invocation.

**Priority rule (v4):** DB config always takes precedence over env vars for operational adapters (LLM, Ticket, Notify, Context, Explainability). If no DB row exists for a key, the env var is the fallback. This allows the system to function correctly on first boot before the UI configures anything.

**Rule:** to add a new provider, write a new adapter class implementing the port and register it in the resolver function inside `infrastructure/container.py`. Do **not** edit nodes, services, the graph, or routes.

## 9. LLM Resilience — Circuit Breaker

`adapters/llm/circuit_breaker.py` wraps any pair of `ILLMProvider` instances:

```
       ┌─────────┐  call ok  ┌─────────┐
       │ PRIMARY │──────────►│ RESULT  │
       └────┬────┘           └─────────┘
            │ N consecutive failures or timeout > T
            ▼
       ┌──────────┐  call ok  ┌─────────┐
       │ FALLBACK │──────────►│ RESULT  │
       └────┬─────┘           └─────────┘
            │ also fails
            ▼
       ┌────────────────────────────┐
       │ DEGRADED MODE              │
       │ Returns deterministic      │
       │ TriageResult(severity=P3,  │
       │  needs_human_review=True)  │
       └────────────────────────────┘
```

State persists in-process. Resets after a configurable cool-down. Every transition emits a structured log event (`triage.fallback_used`, `triage.degraded_mode`) and a Langfuse event.

Configuration (v4 — DB-managed via `ILLMConfigProvider`, env var as fallback):
- `cb_failure_threshold` (default `3`) — in `llm_config` table
- `cb_cooldown_seconds` (default `60`) — in `llm_config` table
- `llm_timeout_s` (default `25`) — in `llm_config` table

**Hot reload support (DEC-A06 / v4):**

The `LLMCircuitBreaker` exposes a `reconfigure(threshold: int, cooldown: int)` method that updates parameters in-place without losing the current failure count or state:
- Parameters update immediately at the next evaluation cycle.
- If the breaker is currently `OPEN`, it remains open until the previous cooldown expires, then uses the new cooldown for subsequent state transitions.
- The `container.reload_llm_adapter()` coroutine (asyncio.Lock-protected) atomically replaces the entire `LLMCircuitBreaker` singleton (including primary and fallback adapters) when the provider or model changes — not just the parameters. This is the correct path when changing providers. `reconfigure()` is the lightweight path when only thresholds change.

## 11. Security Architecture — Prompt Injection Defense (5 layers)

| Layer | What it does | Where |
|---|---|---|
| 1. Input sanitization | Strip control chars, zero-width Unicode. Length limit 8000. File allowlist. Size limit 5MB. | `app/security/input_sanitizer.py` |
| 2. Heuristic detection | Regex against known attack patterns | `app/security/prompt_injection.py` |
| 3. LLM judge | Cheap classification via `ILLMProvider.classify_injection()` | `app/security/prompt_injection.py` |
| 4. Output schema enforcement | Pydantic structured output via `ILLMProvider.triage()`. Free-form text never drives tool calls. | `domain/ports/llm_provider.py` + adapters |
| 5. Tool sandboxing | Adapters accept typed Pydantic args. No shell, no dynamic URLs. | `adapters/**/*.py` |

Blocked incidents are persisted with `blocked=true`, traced with `injection_detected=true`, and shown in the UI as "Blocked by guardrails — see trace".

### 16.2 Route map

| React route | Page component | Auth required | Min role |
|---|---|---|---|
| `/login` | `LoginPage` | No | — |
| `/` | `DashboardPage` | Yes | `viewer` |
| `/incidents/new` | `IncidentFormPage` | Yes | `operator` |
| `/incidents/:id` | `IncidentDetailPage` | Yes | `operator` (own) / `viewer` (all read-only) |
| `/config` | `ConfigPage` | Yes | `flow_configurator` (thresholds only) / `admin` (full) |
| `/admin/users` | `UsersPage` | Yes | `admin` |
| `*` | `NotFoundPage` | No | — |

**Private route behavior:** Unauthenticated users are redirected to `/login`. Users accessing a route they lack role for see a 403 page — they are NOT redirected to login (they are authenticated, just unauthorized).

### 16.3 API contract (React → FastAPI)

| Parameter | Value |
|---|---|
| Base URL | `http://localhost:8000` (dev) / `VITE_API_BASE_URL` env var (prod) |
| Auth header | `Authorization: Bearer {jwt_token}` on all protected requests |
| Token source | `authStore.token` (from `localStorage`) |
| Content-Type | `application/json` |
| CORS | FastAPI `CORSMiddleware` allows origin `http://localhost:5173` (or `CORS_ALLOW_ORIGIN` env var) |

**Error handling contract:**
- HTTP 401 → clear token from `authStore` + `localStorage` → redirect to `/login`
- HTTP 403 → show inline 403 error page (do not redirect)
- HTTP 422 → surface validation errors in the form (field-level if possible)
- HTTP 503 → show "Service unavailable — try again shortly" toast
- Network error → show "Cannot connect to server" toast

**Polling:** The dashboard polls `GET /incidents/{id}/status` every 30 seconds for in-progress incidents (SA-006). Implemented via TanStack Query `refetchInterval`.

### 16.4 Build and serve strategy (DEC-A01)

```dockerfile
# services/sre-web/Dockerfile (multi-stage)
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build          # outputs to /app/dist

FROM nginx:alpine AS serve
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
```

```nginx
# nginx.conf — SPA fallback routing
server {
  listen 80;
  root /usr/share/nginx/html;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;   # all routes → index.html (React Router handles)
  }

  location /health {
    return 200 'ok';
    add_header Content-Type text/plain;
  }
}
```

| **ARC-011** | **GitHub Actions CI**: ruff → mypy → pytest → docker build |
| **ARC-015** | **Prompt registry is mandatory.** All prompts resolved via `PROMPT_REGISTRY`; inline prompts in agents prohibited. Versioned (`<purpose>-vX.Y.Z`) and emitted as `llm.prompt_name` / `llm.prompt_version` spans. |
| **ARC-016** | **Evals are a CI hard gate.** Golden-dataset + LLM-judge runner executes on every PR. PR blocked if `avg_overall_score < 0.70` or `adversarial_recall < 1.0`. |
| **DEC-007** | **Prometheus `/metrics` endpoint** exposed by sre-agent via `prometheus-client`. No auth in dev; basic auth toggled by `METRICS_AUTH_ENABLED` in prod. |
| **DEC-008** | **LLM-as-judge eval pipeline in CI** as the primary quality gate for the agent (replaces ad-hoc unit tests for triage correctness). |
| **ARC-017** | **Governance thresholds are runtime config, not code.** All business thresholds stored in `governance_thresholds` PostgreSQL table; accessed via `IGovernanceProvider` with TTL cache. See §4.11. |
| **ARC-018** | **Every triage result must carry explainability data.** `TriageResult.rag_attribution` must be populated before orchestrator root span closes. Empty list allowed only when `degraded=True`. See §4.12. |
| **ARC-019** | **Online eval is mandatory in production.** `IExplainabilityProvider.post_online_eval()` must fire as background task in production. Best-effort in dev. Memory adapter silently drops. See §4.13. |
| **DEC-009** | **Threshold governance table** — three tiers: CI eval gate (ARCHITECTURE.md + env), circuit breaker threshold (env var), business governance (PostgreSQL + UI). No tier may override another's mechanism. See §4.11. |
| **DEC-A01** | **React SPA deployment: separate `sre-web` container (Nginx).** The React SPA is built by a multi-stage Docker build (node:20-alpine build stage → nginx:alpine serve stage) and served from its own container. FastAPI does NOT serve the React build. Rationale: clean separation of frontend/backend build pipelines, no Node.js tooling in the Python container, standard SPA serving pattern. CORS is enabled in FastAPI for `sre-web`'s origin. Trade-off: one extra host port (5173). Benefit: independent deployability and zero coupling between Python container and Node build tools. |
| **DEC-A02** | **JWT stored in `localStorage`.** For the hackathon mock auth context, `localStorage` is used. Rationale: simpler React implementation, no `SameSite`/`Secure` cookie configuration needed, CSRF immunity. Known trade-off: XSS risk. Mitigated by the fact that this is a demo mock with no real credentials. A production deployment MUST migrate to `httpOnly` cookies. This is documented as SA-003 and ARC-022 enforces that the backend always validates the JWT regardless of storage mechanism. |
| **DEC-A03** | **RBAC enforcement: FastAPI dependencies at the route layer.** The `require_role(*roles)` dependency in `app/api/deps.py` is the single enforcement mechanism. No middleware-per-route pattern (too implicit). No domain-layer role checks (violates ARC-022). Defense-in-depth: UI hides unauthorized sections (UX), backend enforces via `require_role` (security). See §4.15. |
| **DEC-A04** | **Port structure for Config-from-DB: three separate ports.** `IGovernanceProvider` (thresholds only — unchanged), `ILLMConfigProvider` (LLM config — new), `IPlatformConfigProvider` (all other operational config — new). Rationale: single responsibility per port, no breaking change to HU-012 implementation, cleaner separation between governance business logic and infrastructure adapter configuration. One key-value table per port type. See §4.14.2 and §12. |
| **DEC-A05** | **Cold-start seed strategy: idempotent Alembic migration seed.** All three config tables are seeded with documented defaults via `INSERT ... ON CONFLICT DO NOTHING` in the Alembic migration. Runs as part of `alembic upgrade head` at container startup (already in the Docker Compose command). No separate startup check in `container.py`. DB unavailability at startup prevents boot (Docker Compose health check on `app-db`). After boot, all config providers return in-memory defaults on DB error — they MUST NOT raise. See §4.14.3. |
| **DEC-A06** | **LLM hot reload: atomic singleton replacement with asyncio.Lock + circuit breaker reconfigure().** `container.py` holds a singleton `LLMCircuitBreaker` reference behind an `asyncio.Lock`. `reload_llm_adapter()` atomically replaces the singleton when the provider or model changes. `LLMCircuitBreaker.reconfigure(threshold, cooldown)` is the lightweight path for parameter-only changes (preserves failure count). Total propagation SLO: < 5 seconds. See §4.14.4 and §9. |
| **DEC-A07** | **eShopOnWeb integration: new `GithubContextAdapter` + ZIP download.** A new `adapters/context/github_context_adapter.py` implements `IContextProvider`. It downloads the eShopOnWeb repo as a ZIP via HTTPS (no `git` binary needed in Docker image), extracts targeted directories (Controllers, Services, Entities, Interfaces, Infrastructure), chunks and embeds via FAISS. Indexing runs at startup if no cached index exists on the volume. The static FAISS adapter remains as fallback (AC-03 of HU-P030). Pre-built index shipped in the hackathon Docker image (BR-08). |
| **DEC-A08** | **`storage_provider` stays in `.env` (bootstrap only — not in the Config UI).** Changing `storage_provider` to `memory` in a running production system causes data loss on restart. This is an installation parameter, not an operational one. It is classified as BOOTSTRAP and excluded from the `platform_config` UI (although it appears in the DB table as a seed row for documentation purposes, it is not editable via the API). The UI shows a read-only indicator of the current backend. A superadmin who needs to change this must edit `.env` and restart — this is intentional friction. See §14 and HU-P032 AC-13. |
| **DEC-A09** | **`IGovernanceProvider` scope stays unchanged (thresholds only). HU-012 implementation plan is unaffected.** The expansion of "business configuration" introduced by HU-P032 does NOT expand `IGovernanceProvider`. ## 18. Threshold Governance — Where Each Threshold Lives

This table is the authoritative reference for where any threshold in the system MUST be stored. Violating the storage location is an ARC-017 violation.

| Threshold type | Examples | Storage location | Who may change it | Rationale |
|---|---|---|---|---|
| **CI eval gate** | `avg_overall_score >= 0.70`, `adversarial_recall == 1.0` | `ARCHITECTURE.md` §4.5 + env var `EVAL_MIN_SCORE` | `@architect` only | Architectural decision — defines acceptable agent quality. Requires architect approval and doc update. |
| **Circuit breaker** | failure count, cooldown, timeout | env vars (`LLM_CIRCUIT_BREAKER_THRESHOLD`, `LLM_CIRCUIT_BREAKER_COOLDOWN_S`, `LLM_TIMEOUT_S`) | Operator (per environment) | Deployment parameter — may vary between dev/staging/prod without code or architecture changes. |
| **Business governance** | confidence escalation, kill switch, severity routing, quality score, max RAG docs | PostgreSQL `governance_thresholds` + `GET/PUT /governance/thresholds` UI | Operator at runtime via UI | Runtime business decisions — must be changeable without redeploy. Versioned via `updated_at` / `updated_by`. |

**Enforcement rule (ARC-017):** No business governance threshold may be defined as a Python literal, a pydantic-settings field, or an environment variable. The only exception is the `GovernanceThresholds` Pydantic model's field defaults — these serve as safe fallback values when the DB is unreachable, and are not the operational defaults.

**v4 extension — Config-from-DB (ARC-020):** The three-tier model above applies only to governance thresholds. For the full Config-from-DB principle covering ALL business configuration variables, see §4.14. The storage tier for all non-governance business config is `platform_config` or `llm_config` tables (not `governance_thresholds`).
